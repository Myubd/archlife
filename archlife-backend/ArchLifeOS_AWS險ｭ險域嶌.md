# ArchLifeOS バックエンド アーキテクチャ設計書 (プライバシー重視版)

対象: 人生管理OS「ArchLifeOS」（Notion + Google Calendar + 家計簿 + Habitica + Todoist を統合したサービス）
現状のフロントエンド（React, `ArchLifeApp.jsx`）はブラウザ内ストレージで動作するプロトタイプです。本書は、**個人の特定を避け（ログイン不要）**、**AIはまず自己ホストのローカルモデルで処理し、必要な時だけ外部APIを叩く**という方針で、実際のAWSインフラの上に構築するための設計です。

---

## 1. 全体方針

- **ログイン（メールアドレス等の個人情報を伴う認証）を使わない。** 代わりに端末側で生成した匿名ID・暗号鍵でユーザーを区別する。
- **サーバー側は「誰が」使っているか分からない状態を保つ。** データは端末側で暗号化してから送信し、サーバーは暗号文と匿名IDだけを扱う（サーバーが盗まれても中身は読めない）。
- **AIは既定でローカル（自己ホスト）。** Qwenなどのオープンウェイトモデルを自前のサーバーで動かし、通常の「今日の提案」「支出分析」「習慣分析」はそこで完結させる。より高度な分析がしたい時だけ、ユーザーが明示的にオンにしてClaude/GPTの外部APIを呼ぶ。
- 一気に全部作らず、フェーズを分けて実装できる構成にしてある（8章参照）。

---

## 2. 匿名データフロー（認証を使わない設計）

```
端末 (ブラウザ/アプリ)
  匿名IDと暗号鍵を生成
   │  暗号文のみ送信
   ▼
ECS API
  中身を復号せず中継
   │  暗号文 + 匿名IDのみ
   ▼
RDS + S3
  暗号文と匿名IDだけを保持
```
*(図はチャット上部のウィジェットを参照)*

### 仕組み

- 初回起動時、端末側（ブラウザ）でランダムな**匿名ID**（UUID）と、パスフレーズから導出する**暗号鍵**を生成する。
- Todo・日記・家計簿など全データは、端末側で暗号化してからサーバーに送る（クライアントサイド暗号化 / E2E方式）。
- サーバー（ECS・RDS・S3）は暗号文と匿名IDしか持たない。中身は復号できないし、メールアドレスや氏名などの個人情報も一切収集しない。
- 複数端末で同期したい場合は、同じ匿名ID・パスフレーズを他の端末に手入力（またはQRコードで受け渡し）することで同期する。**パスフレーズを忘れると復元できない**ため、その旨をUIで明示する。
- Cognitoのようなユーザープール型の認証サービスは使わない。IPアドレスなど最低限のアクセス制御はALB/WAFのレート制限で対応する。

### この設計のトレードオフ

| メリット | 留意点 |
|---|---|
| メールアドレス等の個人情報を一切保持しない | パスワードリセットのような「本人確認による復旧」ができない（本人確認の手段＝個人情報になるため） |
| サーバーが漏洩してもデータは暗号文のまま | 検索・集計をサーバー側で行えない（暗号化されているため、集計は端末側かLambdaが復号後にメモリ上でのみ行う設計にする） |
| ログイン画面が不要でオンボーディングが速い | 匿名IDを保存した端末を紛失するとパスフレーズなしでは復旧できない |

## 3. 同期処理（CRUD）の全体構成

```
端末 → CloudFront + S3 (フロント配信)
端末 → ALB → ECS Fargate (API) → RDS (暗号文) / ElastiCache (キャッシュ)
```

- CloudFront + S3: Reactの静的ビルドを配信。
- ECS Fargate: 暗号文のまま受け取り、RDSへ保存・取得するだけのAPI（中身を見ない）。
- ElastiCache (Redis): 復号はしないが、「直近アクセスした暗号化ブロブ」をキャッシュしてDB負荷を下げる用途に限定する（平文キャッシュはしない）。

---

## 4. AI推論の2段構成（ローカル優先、外部APIはオプトイン）

```
Lambda (推論先を判定)
   ├─ 既定: ローカルLLM (Qwen等を自己ホスト)
   └─ オンにした時だけ: 外部API (Claude / GPT)
          │
          ▼
   結果保存 & 通知 (RDS + SNS) → ユーザー
```
*(図はチャット中盤のウィジェットを参照)*

### ローカルLLM（既定パス）

- EC2のGPUインスタンス（例: `g4dn.xlarge` や `g5.xlarge`）上で、Ollama や vLLM などを使ってQwen2.5系のモデル（7B〜14B程度が実用的）をホストする。
- このインスタンスはVPC内のプライベートサブネットに置き、外部には公開しない。ECS/Lambdaからのみ到達可能にする。
- 「今日やるべきこと提案」「支出分析」「習慣分析」は、まずこのローカルモデルに投げる。データが外部に一切出ないため、プライバシー上もっとも安全な経路になる。

### 外部API（オプトイン時のみ）

- UI側に「詳しい分析をClaude/GPTに依頼する」のようなトグルを用意し、明示的にオンにした時だけ呼び出す（現在の`AiCard`のボタン方式とも相性が良い＝そもそも1アクション1呼び出しの設計になっている）。
- 呼び出す際は、生の日記本文などをそのまま送らず、**集計済み・匿名化した数値**（例:「食費 ¥32,000、交通 ¥8,000」のようなカテゴリ別合計）だけを送ることでデータ最小化する。
- APIキーはSecrets Managerで管理する。

### 判定ロジック（Lambda）

- ユーザーの設定（端末側に保存、匿名IDに紐づく設定値）を見て、外部APIが有効化されているリクエストのみ外部分岐に振り分ける。既定はローカルLLMのみ。

## 5. 非同期処理全体（イベント駆動）

```
ECS(API) → EventBridge → SQS → Lambda(推論先を判定) → [ローカルLLM / 外部API] → RDS + SNS
```

- ECSはユーザー操作（例: 支出を記録した）をイベントとしてEventBridgeに発行し、即座にレスポンスを返す。
- EventBridgeが種類ごとにルーティングし、SQSにキューイング。
- Lambdaがキューを消費し、上記の2段構成でAI処理を行い、結果をRDSに保存、SNSでユーザーに通知する。

### EventBridgeのスケジュール実行（定期バッチ）

| タイミング | 処理内容 |
|---|---|
| 毎日 朝7時 | 習慣の連続日数（ストリーク）を再計算 |
| 毎日 正午 | サブスクの次回請求日が近いものを検出し通知 |
| 毎月 1日 | 資産推移・支出サマリーを生成し通知 |

いずれも既定ではローカルLLM（または単純な集計ロジック）で完結させ、外部API送信は行わない。

## 6. 機能ドメインとAWSサービスの対応表

| 機能ドメイン | 主な処理 | 使うサービス |
|---|---|---|
| 匿名ID・端末鍵管理 | 端末側で生成・保管（サーバーに個人情報なし） | (サーバー側の対応サービスなし。ECS/RDSは匿名IDのみ扱う) |
| Todo / カレンダー / 習慣 / 日記 / メモ / 目標 | 暗号化CRUD | ECS, RDS |
| 家計簿 / サブスク / 資産推移 | 暗号化CRUD + キャッシュ | ECS, RDS, ElastiCache |
| AI: 今日の提案・支出分析・習慣分析 | 既定でローカルLLM、オプトインで外部API | EventBridge, SQS, Lambda, EC2(GPU, Qwen自己ホスト), SNS |
| 添付ファイル（日記の画像等） | 暗号化アップロード/配信 | S3, CloudFront |
| 通知（リマインド等） | プッシュ/メール配信 | EventBridge(スケジュール), Lambda, SNS |
| フロント配信 | 静的アセット配信 | CloudFront, S3 |

## 7. ネットワーク構成（VPC）とセキュリティ

- ALBのみパブリックサブネットに配置。ECS、RDS、ElastiCache、ローカルLLM用EC2はすべてプライベートサブネットに配置。
- ローカルLLM用EC2インスタンスは外部からのインバウンドを一切許可せず、ECS/Lambdaのセキュリティグループからのみアクセス可能にする。
- 外部API（Claude/GPT）呼び出し時のみ、NATゲートウェイ経由の限定的なアウトバウンドを許可する。
- RDS・S3は保管時暗号化（KMS）を有効化。ただし中身自体もすでに端末側で暗号化済みなので、二重に保護される。
- ログ（CloudWatch Logsなど）にも復号後の個人データが残らないよう、Lambda/ECS側でログ出力時に本文をマスキングする実装を徹底する。

## 8. コストの考え方

- ECS Fargate・RDS・S3・CloudFrontは前回同様、小規模なら月数千円程度から開始可能。
- ローカルLLM用のGPU EC2インスタンス（`g4dn.xlarge`など）が新たなコスト要因になる。常時起動だと月2〜3万円台になり得るため、利用頻度が低いうちは**オンデマンドで起動し、一定時間アイドルなら自動停止する**運用（Lambdaでのスケジュール起動/停止、またはSageMaker Serverless Inferenceの検討）でコストを抑える。
- 外部API（Claude/GPT）はオプトイン時のみの従量課金なので、既定運用では費用がほぼ発生しない。

## 9. 実装フェーズ（一気にやらない前提）

| フェーズ | やること | 使うサービス |
|---|---|---|
| **フェーズ1: 匿名CRUD基盤** | 匿名ID方式のAPIとRDS保存、フロント配信 | ECS, RDS, S3, CloudFront |
| **フェーズ2: 体感速度の改善** | 暗号化ブロブのキャッシュ | ElastiCache |
| **フェーズ3: ローカルAI** | EC2でQwenをホストし、既定のAI機能をローカル化 | EventBridge, SQS, Lambda, EC2(GPU) |
| **フェーズ4: 外部APIオプトイン & 通知** | Claude/GPTのオプトイン呼び出し、リマインド通知 | SNS, Secrets Manager |

この順番なら、フェーズ1〜2の時点で「個人情報を持たないライフ管理OS」として成立し、フェーズ3・4は後から積み増せます。

## 10. 今のプロトタイプ（フロントエンド）との接続点

現状のReact側は`window.storage`（ブラウザ内ストレージ）でデータを持っています。フェーズ1実装後は、`window.storage`への読み書きを「端末側で暗号化 → APIへ送信」「APIから取得 → 端末側で復号」というラッパー関数に置き換えるだけで、UIコンポーネント自体はほぼそのまま流用できます。AIタブの`callClaude`関数は、フェーズ3以降はまずローカルLLMのエンドポイントを叩き、ユーザーが外部APIをオンにしている場合のみ現在のAnthropic API呼び出しにフォールバックする形に拡張します。
